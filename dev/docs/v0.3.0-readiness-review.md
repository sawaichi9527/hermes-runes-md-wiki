# v0.3.0 Readiness Review

Status: draft review
Date: 2026-06-08
Scope: personal-local / open-beta / fresh-clone readiness

## Purpose

`v0.3.0` should not be treated as a simple replacement tag for `v0.1.0-beta.1`.

The version should represent a cleaner external-user baseline after:

- runtime wiki path cleanup
- fresh clone deployment review
- dependency footprint review
- root repository layout cleanup
- developer-only fixtures/history separation
- user support evidence workflow preservation

Do not tag `v0.3.0` until the checks in this file are resolved or explicitly deferred.

## Release Positioning

`v0.3.0` should mean:

```text
Hermes Runes MD Wiki has a cleaner runtime seed, a clearer developer boundary, and a less confusing fresh-clone onboarding path.
```

It should not imply:

- enterprise packaging completeness
- multi-user SaaS readiness
- daemonized memory service readiness
- autonomous memory writer trust
- cloud deployment standardization

The boundary remains:

```text
single-user / local-first / governed personal RAG / developer-friendly but not enterprise-heavy
```

## Current Good State

The previous runtime wiki cleanup is already moving in the right direction:

- `wiki/k6-freelancer` is no longer the runtime-facing default namespace.
- `wiki/sample-project` and `wiki/owner-runes` are no longer public runtime seed targets.
- `wiki/freelancer` is documented as the current dogfood host workspace, not a universal tester default.
- developer history now has an explicit `dev/` boundary.

## Boundary Clarification: User Support Evidence Is Runtime-support, Not Developer-only

Observation, lightweight local logs, and support evidence bundle features must remain available to normal users.

This includes features such as:

```text
observation logs
observation summary
support bundle / issue evidence export
Ragnarok-style diagnostic or log packing workflow
```

Rationale:

- users need a simple way to report problems and reproducible evidence to developers
- developers need observation data to improve prompts, sanitizers, model profiles, and hardcoded heuristics
- keeping support evidence available reduces back-and-forth debugging burden
- this is still personal-local supportability, not enterprise telemetry

Policy:

```text
User-generated runtime logs are local artifacts and should remain gitignored.
Support bundle tooling may remain user-facing.
Developer-only test fixtures and milestone evidence should move under dev/.
```

Do not move or hide user-facing observation / support evidence commands merely because they mention logs.

## Blocking Concern 1: Fresh Clone Dependency Footprint

Fresh deployment currently needs another review because `pip install` may still pull a large PyTorch / NVIDIA CUDA dependency chain when the user only expects a lightweight local Markdown + PostgreSQL / FTS baseline.

### Desired Dependency Profiles

Use split dependency profiles:

```text
requirements-core.txt       Minimal runtime core. No torch, no CUDA, no embedding model stack.
requirements.txt            Safe default alias to core, or minimal aggregator only.
requirements-embedding.txt  Optional local embedding stack. May include sentence-transformers / torch.
requirements-dev.txt        Developer-only test/lint/dev helpers.
```

### Policy

Default quickstart should install only the lightweight core profile:

```bash
pip install -r ../../requirements-core.txt
```

or, if `requirements.txt` is kept as the public default:

```bash
pip install -r ../../requirements.txt
```

then `requirements.txt` must remain lightweight and must not pull torch/CUDA dependencies.

Embedding/vector features should be explicitly optional:

```bash
pip install -r ../../requirements-embedding.txt
```

### Review Commands

Run from a clean clone / clean venv:

```bash
cd ~/workspace/hermes-runes-md-wiki
rm -rf /tmp/hermes-runes-v030-venv
python3 -m venv /tmp/hermes-runes-v030-venv
source /tmp/hermes-runes-v030-venv/bin/activate
python -m pip install --upgrade pip
pip install -r requirements.txt
pip freeze | grep -Ei 'torch|nvidia|cuda|cudnn|triton|sentence|transformers' || true
pip list | wc -l
```

Expected for the default profile:

```text
No torch / nvidia / cuda / cudnn / triton dependency chain.
```

Then separately test optional embedding:

```bash
pip install -r requirements-embedding.txt
pip freeze | grep -Ei 'torch|nvidia|cuda|cudnn|triton|sentence|transformers' || true
```

If CUDA packages appear only here, that is acceptable but should be documented as optional and heavyweight.

These should be reviewed before `v0.3.0`.

Keep user support evidence features reachable from the normal install surface, even if their implementation stores local logs outside git.

## Directory Classification Draft

### Keep at Root / User-facing Surface

```text
bin/
config/
docs/
tools/
wiki/
```

Rationale:

- `bin/` is the stable CLI wrapper surface.
- `config/` may contain runtime-safe default config or examples.
- `docs/` is public tester/runtime documentation.
- `tools/` contains the active importer/retrieval/governance/support implementation.
- `wiki/` is the runtime Markdown memory seed.

### Must Preserve for Normal Users

```text
observation tooling
observation summary tooling
support evidence bundle tooling
Ragnarok-style diagnostic / log packing tooling
```

Rationale:

- these are required for open-beta issue reporting and developer feedback
- users should be able to collect evidence without becoming repository developers
- generated logs and bundles should remain local artifacts and gitignored by default

Possible implementation locations:

```text
tools/importer/               if tightly coupled to memory runtime
tools/support/                if support-bundle logic becomes separate
bin/                          wrapper commands only
```

Avoid placing these only under `dev/`, because that would incorrectly signal they are developer-only.

### Review Before Moving

```text
migrations/
```

Rationale:

- Migrations may be runtime-needed by `bin/hermes-memory-migrate`.
- If moved, wrappers and docs must be updated together.
- A possible future path is `tools/importer/migrations/` rather than `dev/migrations/` if migrations are required by normal users.

Do not blindly move `migrations/` to `dev/` until migration command references are checked.

### Likely Move to `dev/`

```text
db/
fixtures/
reports/
smoke/
templates/
```

Rationale:

- these appear developer/test/evidence oriented
- they should not be visually presented as required fresh-clone runtime directories
- normal users should not need to inspect milestone fixtures or developer reports during installation

Caveat:

```text
reports/ may contain either developer milestone reports or user support bundle examples.
```

If `reports/` contains only generated user evidence output, the output path should be gitignored and documented as local runtime artifact. If it contains checked-in milestone reports, move those under `dev/reports/`.

Before moving, run reference checks:

```bash
grep -RIn "db/\|fixtures/\|reports/\|smoke/\|templates/" \
  AGENTS.md README.md QUICKSTART.md docs wiki bin tools migrations config \
  2>/dev/null | head -200 || true
```

Then update references and smoke wrappers in the same commit.

## Deployment Download Boundary

Long-term, normal users should not need developer-only materials for runtime deployment.

However, avoid introducing complex packaging too early.

Preferred near-term approach:

1. Keep one GitHub repository.
2. Move developer-only materials under `dev/`.
3. Keep observation and support evidence commands available to normal users.
4. Document that `dev/` is not runtime memory and not imported by default.
5. Later consider release archive packaging or sparse checkout only if repository size becomes a real burden.

Avoid adding enterprise-grade packaging systems at this stage.
